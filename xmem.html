from xmem import XMem

mem = XMem(api_key="xm_...")

# 1. write a fact (or an episode)
mem.write(
  agent_id="support-bot",

ctx = mem.recall(

# 3. inject into your LLM call
llm.complete(system=ctx.prompt, ...)
The problem

Your agent remembers everything. That's the bug.

Stuff the last N turns into context. Bolt on a vector DB. Ship. Six weeks later your agent confidently tells a user their plan is "Pro" when they downgraded three messages ago. You ship a patch. It breaks something else.

Memory for agents isn't a retrieval problem. It's a belief problem — which facts are still true, which got superseded, and which source you trust when they disagree.

A traditional vector DB stores strings. XMem stores beliefs — with lineage, supersedes, and decay.

Under the hood

Why not just a vector DB?

A vector DB answers "what looks like this?" A memory engine answers "what's true right now?" Those are different queries, and they need different data structures.

Dimension
Vector DB + RAG
"what looks like this?"
XMem
"what's true right now?"
What it stores
✕Embeddings of strings. No notion of "true right now."
✓Structured facts with timestamps, sources, and confidence.
Conflicts
✕Contradictions coexist forever. Retrieval picks whichever scores higher.
✓Newer facts supersede older ones. Retractions are first-class.
Lineage
✕None. You can't answer why do you believe this?
✓Every answer carries lineage. One call returns the fact + its provenance.
Time decay
✕None. A fact from Jan 2024 ranks with a fact from yesterday.
✓Decay curves per fact type. Preferences age differently than contracts.
Identity
✕"Lisa" and "L. Chen" are strangers.
✓Entity resolution built in. "Lisa" = "L. Chen" = user:u_4713.
Integration
✕You'll end up writing a belief layer on top.
✓One SDK call. Works alongside your existing vector DB — we don't replace it.

Core primitives

Six objects. One coherent mental model.

Everything in XMem is built from these. Learn them in an afternoon; never hit a dead end.

Fact

A structured belief about a user, entity, or world state. Has content, confidence, source, and a decay curve.

mem.write(content=...)

Episode

A conversation, session, or interaction — raw. XMem extracts facts from episodes automatically.

mem.log_episode(...)

Artifact

Structured objects your agent produces — tickets, PRs, summaries. Linked back to the facts that shaped them.

mem.attach_artifact(...)

Supersede chain

New facts supersede old. The chain is queryable — you can always replay the belief history for any entity.

mem.history(fact_id=...)

Policy

Who (which agent, which user tier) sees what. Scoped at fact-level. Useful when one engine serves multiple products.

mem.policy.grant(...)

Audit log

Every read, write, and supersede is logged with actor and timestamp. Export to your SIEM. Enterprise-grade, day one.

mem.audit.stream(...)

FAQ

Developer questions.

How does XMem relate to Memory Hub? +

Same engine, two surfaces. Memory Hub is the product your company's team uses. XMem is the API your agents use. You can run both against the same memory store.

Do I still need my vector DB? +

If you're doing document RAG, keep it. XMem handles belief state — user preferences, decisions, entity resolution, supersedes. Most teams end up running both.

What's the latency overhead? +

p50 under 80ms on Scale, p99 under 220ms. Fast enough for voice. We publish full latency distributions on the status page.

Can I self-host? +

Enterprise customers get a single-tenant VPC deploy. We don't publish a self-host-anywhere image yet — that's on the roadmap for H2 2026.

What LLMs does XMem work with? +

All of them. XMem returns a context string and structured facts. You pass them into OpenAI, Anthropic, Gemini, Llama, Mistral — whatever you're using.

How do you handle PII? +

Tagged at the fact level, encrypted at rest, scoped via policy. Delete-by-user-id is one API call and cascades through the entire supersede chain.

Is it open source? +

Yes — the SDK is MIT-licensed at github.com/XTraceAI/xtrace-sdk, including our homomorphically encrypted vector store built from the ground up. The belief-revision engine is commercial — that's how we fund the research.
